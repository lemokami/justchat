Building a universal ACP client

From a one-line idea — “a desktop UI for any coding agent” — to a packaged app. The decisions, the dead-ends, and the things still on the to-do list.

Engineering notes · Rust · Tauri · Vue · Agent Client Protocol

The goal: a polished, app-like chat client that can talk to any agent speaking the Agent Client Protocol (ACP) — the JSON-RPC 2.0 protocol coding agents like kiro-cli, Gemini CLI, and Claude Code expose over stdio. You bring the agent and your API keys; the app gives you streaming chat, tools, approvals, history, and attachments.

The core design principle

One decision shaped everything: keep the agent engine completely separate from the UI, connected by a single seam — two channels carrying plain, serializable enums:

  • Command (UI → engine): CreateSession, SendPrompt, Cancel, PermissionDecision, SetModel, Connect/Disconnect.
  • Event (engine → UI): MessageChunk, ThoughtChunk, ToolCall, PermissionRequested, TurnEnded, and friends.

The engine (acpc_protocol) knows nothing about the UI; the UI knows nothing about ACP. That seam is the reason we could swap the entire frontend later without touching the engine.

The journey

1

The headless engine first

Before any pixels, we built and tested the protocol layer: spawn kiro-cli acp, do the ACP initialize handshake, create a session, send a prompt, and stream session/update notifications back. A scripted mock agent let us unit-test the whole flow — handshake, prompts, tool calls, the permission loop, even crash detection — without a UI or the real agent.

2

GPUI: the first UI

The first frontend used GPUI (Zed’s GPU-accelerated Rust UI). It worked — native, fast — but came with friction (see challenges below). Crucially, because of the Command/Event seam, the UI was just one consumer of the engine.

3

Pivot to Tauri + Vue

We moved to Tauri with a Vue 3 webview. The Rust engine stayed identical; we added a thin Tauri backend that maps invoke() calls to Commands and forwards Events to the webview. First as hand-rolled Vue + Tailwind, then re-built on the AI Elements Vue component library (shadcn-vue + Tailwind 4) for a true AI-chat look: conversation, message bubbles, streaming Markdown, collapsible reasoning, tool cards.

4

From client to platform

Then the features that make it a real product: configurable agent profiles (any command/args/env + API keys, with presets), one-click connect/disconnect, persistent chat history with editable titles, attachments, model selection, and slash commands surfaced from the agent. Finally, packaging into a .dmg and a CI workflow for multi-OS installers.

Hard problems & how we solved them

1. The ACP crate assumed integer JSON-RPC ids

kiro-cli sends agent→client requests (permission, file reads, terminals) with string ids; the pinned agent-client-protocol 0.4.x assumed integers, so those requests failed to parse and were silently dropped — the agent hung forever waiting for a permission reply. The fix is a transparent id-compat shim on the byte stream: rewrite incoming string ids to synthesized integers (remembering the mapping), and translate our outgoing responses back. Numeric-id agents pass through untouched.

2. !Send futures

The ACP connection’s futures are !Send, so they can’t move across threads. We run them on a dedicated OS thread with a current-thread Tokio runtime + LocalSet, and bridge to the UI thread only through the serializable channel API.

3. macOS Metal toolchain

Both GPUI and Tauri compile Metal shaders at build time; Xcode 26 ships the Metal compiler as a separate component. Building failed until we ran xcodebuild -downloadComponent MetalToolchain.

4. No Node — then Node

The environment had no Node, so the first Vue UI ran from vendored global builds (Vue + Tailwind via CDN files, no bundler). To adopt AI Elements Vue (a shadcn-vue registry that needs Vite + Tailwind 4), we installed Node and scaffolded a proper Vite project, wiring its built dist/ into Tauri’s frontendDist.

5. The “it knows my project” surprise

Because the agent was launched in the project directory, it read the repo and greeted us with project context. For a general chat app, a new chat should start clean — so sessions now use an empty ~/.acp-chatbot/workspace by default (overridable per agent).

6. Bundled-app PATH

A macOS .app launched from Finder gets a minimal PATH and can’t find user-installed CLIs (kiro-cli, npx, gemini). The backend now augments the spawned agent’s PATH with ~/.local/bin, /opt/homebrew/bin, etc.

Smaller fixes

  • Thought text rendered one word per line — we were inserting newlines between streamed chunks; fixed by concatenating verbatim.
  • Long tool commands overflowed the window — a classic flexbox min-width issue; fixed with min-w-0 + wrapping.
  • Tool cards stuck on “pending” — finalized lingering statuses when a turn ends.

Current limitations

These are known and on the roadmap — worth being upfront about.
  • No true context resume. Reopening a saved chat shows the full transcript, but the agent’s memory isn’t restored — a new prompt starts a fresh ACP session. Real resume needs ACP session/load against the agent’s own session store.
  • Unsigned builds. The distributed .dmg isn’t codesigned/notarized, so macOS Gatekeeper requires right-click → Open. Proper distribution needs an Apple Developer ID.
  • Auth is env-only. Agents that authenticate via API-key env vars work; agents requiring an interactive ACP authenticate (OAuth-style) flow aren’t wired to a UI yet.
  • Agent must be installed. JustChat is a client; it spawns an agent you’ve installed and put on PATH. A missing binary surfaces a clear “failed to start agent”.
  • Images sent inline. Image attachments are base64-encoded into the prompt, which inflates large payloads; other files are sent as file:// resource links.
  • Slash commands depend on the agent. The / menu lists whatever the agent advertises via ACP; if it advertises none, the menu is empty.
  • Per-OS builds. Tauri can’t cross-compile; installers are produced on each OS (handled by the CI workflow on tag).

Try it

# run from source
cd crates/acpc_app/ui && npm install && npm run build
cd ../../.. && cargo run -p acpc_app

# build an installer (macOS)
cd crates/acpc_app/ui && npm run build
cd .. && ui/node_modules/.bin/tauri build   # → target/release/bundle/
