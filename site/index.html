A native desktop client for any ACP agent

Bring your own coding agent (kiro-cli, Gemini CLI, Claude Code, or a custom one) and your own API keys, and chat through a polished, app-like UI. A tiny Rust engine speaks the Agent Client Protocol over stdio; a Tauri + Vue frontend renders it.

What it is

JustChat is a client for the Agent Client Protocol (ACP) — a JSON-RPC 2.0 protocol (the same one Zed uses) that lets a UI drive any compliant coding agent running as a subprocess.

Bring your own agent

Pluggable

Configure command, args, working dir and per-agent env vars (API keys). Presets for Kiro, Gemini, Claude Code. Connect / disconnect in one click.

App-like UX

Modern chat

Streaming, Markdown + syntax highlighting, collapsible reasoning, tool-call cards, inline approval prompts, slash commands, attachments, model picker.

Local-first

Persistent

Chats saved to disk, listed in the sidebar, reopenable across restarts, with editable titles. Agents and API keys stored locally.

Architecture

A Cargo workspace with a strict split: the protocol engine is UI-agnostic and fully tested without any UI.

CrateRole
acpc_protocolHeadless ACP engine — subprocess management, the ACP Client impl, a thread bridge exposing a serializable Command/Event channel API, a JSON-RPC id-compat shim, settings, and fs/terminal handlers.
acpc_coreFramework-independent state machine + Markdown/syntax helpers (pure logic, unit-tested).
acpc_appThe Tauri app: Rust backend (commands, event forwarding, agent profiles, chat persistence) + the Vue frontend under ui/.
┌─ Webview (Vue 3 + AI Elements) ─┐  invoke()   ┌─ Tauri backend (Rust) ─┐  stdio  ┌────────────┐
│  chat UI · agents · history     │ ─────────►  │  acpc_protocol engine  │ ──────► │  ACP agent │
│                                 │ ◄─────────  │  (bridge + id-shim)    │ ◄────── │ subprocess │
└─────────────────────────────────┘  acp-event  └────────────────────────┘         └────────────┘
The UI thread and the protocol thread share nothing but two channels carrying plain Command (UI→engine) and Event (engine→UI) enums. That single seam is why the same engine drove a GPUI prototype, headless examples, and now Tauri.

How a turn works

UI → engine

Commands

CreateSession, SendPrompt, Cancel, PermissionDecision, SetModel, Connect/Disconnect.

engine → UI

Events

Connected, SessionCreated, MessageChunk, ThoughtChunk, ToolCall(Update), PermissionRequested, TurnEnded, ModelsAvailable, CommandsAvailable, Error, Disconnected.

  1. You type and hit Enter → the Vue app pushes a user message and a streaming placeholder, then invoke("send_prompt").
  2. The Tauri backend forwards a SendPrompt command to the engine thread, which calls session/prompt over stdio.
  3. The agent streams session/update notifications (thoughts, text, tool calls). The engine maps each into an Event and forwards it to the webview as acp-event.
  4. If a tool needs approval, the agent calls session/request_permission → surfaces as a card with Allow/Reject (or auto-approves per policy).
  5. The turn ends; the transcript is written to ~/.acp-chatbot/chats/.
